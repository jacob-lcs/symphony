# Symphony 项目说明与 Harness Engineering 意义

## Symphony 是什么？

Symphony 是一个长期运行的自动化服务，它能够将项目工作转化为独立的、自主的实现运行，使团队能够管理工作本身，而不是监督编码代理（coding agents）。

### 核心功能

Symphony 的主要功能包括：

1. **自动化任务编排**：持续从问题跟踪系统（如 Linear）读取工作任务
2. **隔离工作空间**：为每个问题创建独立的工作空间
3. **代理自动执行**：在工作空间内运行编码代理会话来完成任务
4. **工作流管理**：通过仓库内的 `WORKFLOW.md` 文件管理代理提示和运行时设置
5. **可观测性**：提供足够的可观测性来操作和调试多个并发代理运行

### 工作原理

```
Linear 问题跟踪 → Symphony 轮询 → 创建工作空间 → 启动 Codex 代理 → 执行工作 → 完成任务
```

具体流程：
1. 轮询 Linear 获取待处理工作
2. 为每个问题创建独立工作空间
3. 在工作空间内以 App Server 模式启动 Codex
4. 向 Codex 发送工作流提示
5. 保持 Codex 工作直到任务完成
6. 如果问题状态变为终态（Done、Closed、Cancelled 或 Duplicate），自动停止代理并清理工作空间

## 对 Harness Engineering 的意义

### 什么是 Harness Engineering？

Harness Engineering（驾驭工程）是一门新兴的软件开发学科，专注于让 AI 编码代理能够可靠、高速地构建、测试和维护大规模软件系统。它的核心理念是：**人类工程师不再直接编写代码，而是设计环境、定义架构约束、指定意图，并构建反馈循环，让代理能够安全且高效地生成和维护代码**。

### Harness Engineering 的核心组件

1. **约束（Constraints）**：定义代理可以和不可以做什么，包括架构边界、依赖规则和工作流顺序
2. **上下文与文档（Context & Documentation）**：构建面向代理的文档，帮助代理理解代码库、需求和期望
3. **反馈机制（Feedback Mechanisms）**：集成可观测性工具、linters、结构化测试，让代理能够自主测试、修复和改进输出
4. **自我修复（Self-repair）**：自动化流程定期扫描不一致性，由代理自动修复问题
5. **委派结构（Delegation Structure）**：人类工程师"管理"代理——设定目标、识别失败、调整系统约束

### Symphony 在 Harness Engineering 中的关键作用

Symphony 体现了 Harness Engineering 的核心原则，并为其提供了实践框架：

#### 1. **从监督代理到管理工作的转变**

传统方式：工程师需要持续监督和指导编码代理
```
工程师 → 手动启动代理 → 监督执行 → 处理错误 → 重新运行
```

Symphony 方式：工程师管理工作流程，代理自主执行
```
工程师 → 配置工作流 → Symphony 自动编排 → 代理自主完成 → 提供工作证明
```

#### 2. **提供生产级的代理编排基础设施**

Symphony 解决了四个关键的运营问题：

- **可重复性**：将问题执行转变为可重复的守护进程工作流，而非手动脚本
- **隔离性**：在每个问题的工作空间中隔离代理执行，确保代理命令仅在专用目录内运行
- **版本控制**：将工作流策略保存在仓库内（`WORKFLOW.md`），团队可以对代理提示和运行时设置进行版本控制
- **可观测性**：提供足够的可观测性来操作和调试多个并发代理运行

#### 3. **实现"约束即代码"（Constraints as Code）**

通过 `WORKFLOW.md` 配置文件，团队可以：

```yaml
agent:
  max_concurrent_agents: 10  # 并发限制
  max_turns: 20             # 单次运行的最大轮次
codex:
  approval_policy: {...}     # 审批策略
  thread_sandbox: workspace-write  # 沙箱限制
  turn_sandbox_policy: {...}  # 每轮沙箱策略
```

这些配置定义了代理的运行边界，确保代理在安全的约束下工作。

#### 4. **支持持续的反馈循环**

Symphony 提供了完整的反馈机制：

- **实时日志**：结构化日志记录所有代理活动
- **状态监控**：Web 仪表板和 API 端点实时展示运行状态
- **工作证明**：CI 状态、PR 审查反馈、复杂度分析、演练视频
- **自动重试**：对瞬态故障进行指数退避重试
- **状态同步**：当问题状态变化时自动停止不符合条件的运行

#### 5. **降低运营复杂度**

OpenAI 的 Harness Engineering 实验表明，一个 3-7 人的团队能够在不手写任何代码的情况下交付约 100 万行代码的内部产品。Symphony 通过以下方式使这成为可能：

- **自动化编排**：无需手动管理多个代理会话
- **工作空间管理**：自动创建、维护和清理工作空间
- **状态协调**：自动处理问题状态变化和代理生命周期
- **并发控制**：智能管理多个并发代理运行
- **错误恢复**：自动从瞬态故障中恢复

#### 6. **可扩展和可定制**

Symphony 提供了灵活的扩展点：

- **自定义工作流**：通过 `WORKFLOW.md` 定义特定于项目的工作流
- **生命周期钩子**：`after_create`、`before_run` 等钩子支持自定义设置
- **技能系统**：通过 `.codex/skills/` 目录提供自定义工具（commit、push、pull、land、linear 等）
- **多语言实现**：基于 `SPEC.md` 规范，可用任何语言实现

### Symphony 与 Harness Engineering 的关系总结

| Harness Engineering 原则 | Symphony 的实现 |
|------------------------|----------------|
| 环境设计 | 工作空间管理、沙箱策略 |
| 约束定义 | WORKFLOW.md 配置、审批策略 |
| 反馈循环 | 日志系统、状态监控、CI 集成 |
| 可观测性 | Web 仪表板、API 端点、结构化日志 |
| 自动化编排 | 轮询调度、并发控制、错误恢复 |
| 版本控制 | 仓库内工作流、技能版本化 |
| 人类角色转变 | 从监督代理到管理工作 |

## 实践意义

### 对团队的价值

1. **提高生产力**：工程师可以专注于高层次的工作管理，而非低层次的代码编写
2. **降低认知负担**：无需持续监督代理，系统自动处理执行细节
3. **提高可靠性**：标准化的编排流程减少人为错误
4. **更好的可追溯性**：所有代理活动都有完整的日志和状态记录
5. **易于扩展**：可以轻松并发运行多个代理处理不同任务

### 适用场景

Symphony 最适合：

- 已采用 Harness Engineering 实践的代码库
- 需要大规模自动化开发任务的团队
- 希望从"监督代理"转向"管理工作"的组织
- 需要可靠的代理编排基础设施的项目

### 前置要求

根据文档，Symphony 在以下代码库中效果最好：

1. **已采用 Harness Engineering**：代码库已经为代理工作做好准备
2. **良好的文档**：有清晰的文档帮助代理理解代码
3. **自动化测试**：有完整的测试套件验证代码更改
4. **明确的架构约束**：有清晰的架构边界和规则

## 技术架构

Symphony 包含以下主要组件：

1. **Workflow Loader**：读取和解析 `WORKFLOW.md`
2. **Config Layer**：提供类型化的配置访问器和验证
3. **Issue Tracker Client**：与问题跟踪系统（Linear）通信
4. **Orchestrator**：拥有轮询周期和运行时状态，决定调度、重试、停止
5. **Workspace Manager**：管理工作空间路径、生命周期和清理
6. **Agent Runner**：创建工作空间、构建提示、启动代理、流式传输更新
7. **Status Surface**：提供人类可读的运行时状态展示

## 总结

**Symphony 是 Harness Engineering 从理论到实践的桥梁**。它不仅体现了 Harness Engineering 的核心原则——通过系统设计而非直接编码来驾驭 AI 代理，还提供了一个可运行、可扩展的参考实现。

通过 Symphony，团队可以：
- 将工程师的角色从"代码编写者"转变为"系统设计师和代理管理者"
- 建立生产级的代理编排基础设施
- 实现从监督单个代理到管理整个工作流程的转变
- 在安全的约束下大规模运行 AI 代理

这正是 Harness Engineering 的核心目标：**不是让 AI 更聪明，而是构建让 AI 能够可靠、安全、大规模工作的系统**。
